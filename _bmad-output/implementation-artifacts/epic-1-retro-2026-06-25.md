## Epic Review

Epic 1 dejo la base operativa de Smart Ant: monorepo, API Express, mobile Expo, navegacion principal, registro, login/logout, sesion opaca y contexto financiero inicial por usuario.

Historias revisadas:

- 1.1 Inicializar Smart Ant: review
- 1.2 Mostrar identidad y navegacion base: review
- 1.3 Crear una cuenta: review
- 1.4 Iniciar y cerrar sesion: review
- 1.5 Configurar contexto financiero: review

Nota de estado: las historias estan completas desde desarrollo y validacion local, pero permanecen en `review` hasta aceptacion humana. Esta retrospectiva cierra el pendiente de proceso del epic sin mover esas historias a `done`.

## Lo Que Funciono

- La arquitectura minima aguanto bien: workspaces, API, mobile, Prisma y pruebas quedaron conectados sin introducir capas innecesarias.
- La autenticacion quedo con contratos seguros desde el inicio: Argon2id, sesiones opacas, hash SHA-256 del token persistido y errores genericos para credenciales invalidas.
- La app mobile ya tiene flujo real de usuario: splash, tabs, registro, login, restauracion de sesion, perfil y logout.
- Los gates se ejecutaron durante el epic y quedaron como disciplina base: type-check, lint, tests y build.
- La cuenta demo quedo documentada y disponible para probar la app sin crear usuarios manualmente.

## Dificultades Y Ajustes

- El entorno local no coincide completamente con el objetivo de runtime: el proyecto fija Node.js 24 para CI/produccion, mientras el desarrollo local detecto Node.js 22. La documentacion debe seguir marcando esa diferencia.
- Expo Go necesita que la API sea accesible por LAN; se agrego el flujo `dev:lan` despues para evitar errores de red durante pruebas en dispositivo fisico.
- Varias validaciones tuvieron que separarse en funciones puras para que Vitest no dependiera de modulos nativos de React Native.
- El estado `review` en todas las historias obliga a distinguir entre "implementado y verificado localmente" y "aceptado como terminado".

## Deuda Y Riesgos

- Produccion sigue intencionalmente diferida: credenciales reales, Railway, EAS/APK y variables finales se resolveran cuando se active el paso a produccion.
- El mecanismo temporal de archivos locales para imagenes debe migrar a una estrategia adecuada de produccion antes de EAS/APK.
- Falta aceptacion humana formal de las historias para moverlas de `review` a `done`.
- La verificacion en dispositivo fisico depende de levantar API y Expo con la IP LAN correcta.

## Aprendizajes

- Mantener contratos pequenos por historia redujo retrabajo: registro no hizo login, login no hizo perfil, perfil no hizo calculos financieros.
- Las sesiones opacas fueron suficientes para el MVP y evitan meter JWT, refresh tokens o roles antes de necesitarlos.
- Centralizar el cliente API mobile y SecureStore simplifico registro, login, restauracion y logout.
- Los documentos temporales son utiles, pero deben marcar claramente que no bloquean tareas que dependen de decisiones finales de produccion.

## Preparacion Para Epic 2

Epic 2 entra sobre una base usable: usuario autenticado, perfil financiero con moneda/zona horaria y API protegida. Para que ingresos, salario recurrente, presupuestos y metas no se atasquen, conviene mantener estos acuerdos:

- Seguir derivando `userId` desde la sesion, nunca desde body o URL.
- Crear modelos de dominio de forma aditiva y acotada a cada historia.
- Reutilizar validacion Zod, errores API y cliente mobile existentes.
- Mantener estados accesibles de carga, error y exito en formularios mobile.
- No introducir infraestructura de produccion hasta que se active el gate de Railway/EAS/APK.
