# Epic 2 Retrospective: Plan financiero basico

Status: done
Date: 2026-06-25

## Epic Review

Epic 2 agrego la primera capa real de planificacion financiera: ingresos manuales, salario recurrente, presupuestos y metas de ahorro. El trabajo se construyo sobre la autenticacion y el perfil financiero del Epic 1.

Historias revisadas:

- 2.1 Gestionar ingresos: review
- 2.2 Configurar salario recurrente: review
- 2.3 Crear presupuestos: review
- 2.4 Crear metas de ahorro: review

Nota de estado: las historias estan implementadas y verificadas localmente, pero permanecen en `review` hasta aceptacion humana.

## Lo Que Funciono

- El patron de rutas protegidas por sesion se reutilizo sin redisenar autenticacion.
- Los importes se manejaron como enteros/BigInt en API y strings en contratos, evitando perdida de precision.
- Cada modelo entro con migracion aditiva y alcance claro por usuario autenticado.
- La pantalla Plan pudo concentrar salario, presupuestos y metas sin crear navegacion adicional prematura.
- El salario recurrente quedo idempotente por periodo mediante `SalaryRun`, evitando duplicados al reintentar.

## Dificultades Y Ajustes

- La UI de Plan empezo a acumular varias responsabilidades. Aun sirve para MVP, pero puede necesitar separacion si crecen acciones, filtros o edicion completa.
- Presupuestos y metas comparten patrones parecidos; por ahora se mantuvieron separados para evitar abstracciones prematuras.
- La recurrencia de salario quedo explicita, sin scheduler. Es correcto para el MVP, pero debe quedar claro en producto: el usuario genera el ingreso, no el sistema automaticamente.
- La edicion completa no quedo igual de profunda en todas las superficies mobile; el contrato API cubre mas que algunos cortes de UI.

## Deuda Y Riesgos

- La aceptacion humana sigue pendiente para pasar historias de `review` a `done`.
- Antes de depender de calculos de saldo gastable, hay que confirmar que presupuestos, metas y salario representan bien los casos reales del usuario.
- El Plan puede volverse dificil de mantener si Epic 3 agrega alertas y calculos sin ordenar componentes.
- Produccion sigue diferida; no bloquear estas historias por credenciales finales, Railway o EAS.

## Aprendizajes

- Los modelos de dominio deben nacer con ownership por usuario desde el primer commit.
- La idempotencia debe definirse donde hay generacion de movimientos financieros, no despues.
- Mantener la generacion de salario como accion explicita redujo infraestructura y riesgo operativo.
- No abstraer presupuestos y metas todavia fue correcto: son parecidos, pero sus reglas de producto pueden divergir.

## Preparacion Para Epic 3

Epic 3 necesita calcular saldo gastable, evaluar gastos, mostrar alertas y confirmar movimientos. Para entrar bien:

- Reutilizar ingresos, salario, presupuestos y metas como fuentes de verdad.
- Definir funciones puras de calculo en `packages/finance` o modulo equivalente antes de acoplar UI.
- Mantener importes como enteros y contratos string hacia mobile.
- Cubrir casos borde de periodo semanal/mensual usando la zona horaria del perfil.
- Evitar que la UI de Plan absorba responsabilidades de evaluacion de gastos.

## Action Items

- Owner: Dev. Mantener idempotencia obligatoria para cualquier generacion automatica o semiautomatica de movimientos.
- Owner: Product/QA. Validar manualmente que salario, presupuestos y metas reflejan el flujo esperado antes de mover historias 2.1 a 2.4 a `done`.
- Owner: Dev. Documentar cualquier diferencia entre capacidades API y UI mobile si se decide dejar cortes parciales para MVP.
