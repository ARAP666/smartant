# Epic 3 Retrospective: Evaluacion y control de gastos

Status: done
Date: 2026-06-25

## Epic Review

Epic 3 convirtio la planificacion financiera en decisiones accionables: calculo de saldo gastable, evaluacion de gastos, alertas, confirmacion idempotente y correccion de gastos confirmados.

Historias revisadas:

- 3.1 Calcular saldo gastable: review
- 3.2 Evaluar un gasto manual: review
- 3.3 Mostrar alertas financieras: review
- 3.4 Confirmar un gasto: review
- 3.5 Editar o eliminar gastos: review

Nota de estado: las historias estan implementadas y verificadas localmente, pero permanecen en `review` hasta aceptacion humana.

## Lo Que Funciono

- El motor financiero quedo puro en `packages/finance`, sin dependencia de Expo, Express ni Prisma.
- La regla conservadora de saldo gastable quedo probada: saldo base, presupuestos y metas compiten y gana la restriccion mas baja.
- La evaluacion de gasto no confirma automaticamente; crea movimiento pendiente y permite explicar impacto antes de persistir gasto real.
- Las alertas se derivan del calculo, sin tabla nueva ni promesas de asesoria financiera.
- La confirmacion de gastos usa idempotencia para evitar duplicados por reintentos.
- La edicion y eliminacion protegen ownership por `userId` y usan transacciones para preservar consistencia.

## Dificultades Y Ajustes

- El flujo Add acumulo evaluar, alertar, confirmar, editar y eliminar. Sirve para MVP, pero es el punto con mayor riesgo de complejidad UI.
- La diferencia entre `PendingMovement` y `Expense` debe mantenerse clara para no mezclar intencion con historial real.
- Las alertas requieren lenguaje cuidadoso: explican restricciones del plan, no dan consejo financiero.
- La edicion de gasto obliga a recalcular antes de persistir; esto es correcto pero aumenta superficie de pruebas.

## Deuda Y Riesgos

- Falta aceptacion humana para pasar 3.1 a 3.5 de `review` a `done`.
- La UI de Add puede requerir separacion de componentes antes de agregar OCR o importaciones mas complejas.
- Hay que validar manualmente casos reales con categorias, presupuestos generales, presupuestos por categoria y metas simultaneas.
- Cualquier cambio futuro al motor financiero puede afectar gasto manual, alertas, confirmacion, historial e importaciones.

## Aprendizajes

- El calculo financiero pertenece a un paquete puro; eso hace que API y mobile sean consumidores, no duenos de reglas.
- La idempotencia no es opcional en confirmaciones de dinero.
- Mantener movimientos pendientes antes de gastos confirmados dio un flujo claro para evaluar, explicar y decidir.
- El ownership por usuario debe comprobarse como parte de cada mutacion, no solo en listados.

## Preparacion Para Epic 4

Epic 4 depende de que los datos confirmados puedan resumirse y filtrarse correctamente. Para entrar bien:

- Usar `Expense` confirmado como fuente para historial y categorias.
- Reutilizar el motor financiero solo donde aporte calculo; no mezclarlo con queries de historial.
- Mantener periodos consistentes con `timeZone` del perfil.
- Probar filtros por usuario para evitar fugas de historial.
- Considerar extraccion de componentes en Add si Epic 4 comparte patrones visuales con Movimientos.
